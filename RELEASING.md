# Releasing Cards Central

This is the maintainer checklist for shipping a new version. For day-to-day
contributing (adding shops, running the app/tests) see
[CONTRIBUTING.md](CONTRIBUTING.md).

## How releases flow

A release is driven by **a git tag** and **a published GitHub Release**, which
trigger two independent pipelines:

| Trigger | Workflow | What it ships |
|---------|----------|---------------|
| Push a `v*` tag | [`build-deploy.yml`](.github/workflows/build-deploy.yml) | Native apps — EAS builds iOS + Android and submits to **TestFlight** and the **Play Store internal track** |
| Publish a GitHub **Release** | [`notify-pages.yml`](.github/workflows/notify-pages.yml) | Web — rebuilds **Prod PWA** (`/app/`) and the **landing page** (`/`) on the [`cardscentral.github.io`](https://github.com/cardscentral/cardscentral.github.io) repo |
| Push to `main` | [`notify-pages.yml`](.github/workflows/notify-pages.yml) | Web — refreshes **QA** (`/qa/`) only; no tag/release needed |

So a full release is: **bump the version → push the tag → publish the Release.**

## Versioning

We use [Semantic Versioning](https://semver.org/): `MAJOR.MINOR.PATCH`, tagged
as `vX.Y.Z`.

- **PATCH** — bug fixes, new shops, copy/translation tweaks
- **MINOR** — new user-facing features (backwards compatible)
- **MAJOR** — breaking changes to stored data / share format

Before tagging, bump the version in **all** the places that carry it so the
app, stores and release notes agree:

- `package.json` → `version`
- `app.json` / `app.config.js` → `expo.version` (and bump the native build
  numbers: `expo.ios.buildNumber` and `expo.android.versionCode`)
- `src/screens/SettingsScreen.tsx` → `APP_VERSION` (drives the in-app version
  row and its "release notes" link)

## Step-by-step

> **Workflow note:** this project currently commits **directly to `main`**
> (no pull requests). The steps below reflect that. Because GitHub's
> auto-generated release notes group *merged PRs*, with a direct-commit flow
> the notes fall back to a flat list of commits since the last tag — so keep
> commit messages clear, and edit the release notes by hand before publishing.
> The category config in [`.github/release.yml`](.github/release.yml) still
> applies automatically if/when you switch to PRs later.

1. **Make sure `main` is green.** All checks and E2E suites should pass, and QA
   (`https://cardscentral.github.io/qa/`) should look right.

2. **Bump the version** in the files listed above and commit it directly to
   `main`:

   ```bash
   git checkout main && git pull
   # edit package.json / app.json / SettingsScreen.tsx ...
   git commit -am "chore: release v1.2.3"
   git push origin main
   ```

3. **Tag the release commit** and push the tag — this kicks off the native
   build + store submission:

   ```bash
   git tag v1.2.3
   git push origin v1.2.3
   ```

4. **Publish the GitHub Release** for that tag — this rebuilds the Prod web app
   and the landing page:

   ```bash
   gh release create v1.2.3 --title "v1.2.3" --generate-notes
   ```

   (Or use the GitHub UI: **Releases → Draft a new release → choose the tag →
   Generate release notes → Publish**.) With the direct-commit flow the
   generated notes are a commit list — review and rewrite them into a readable
   summary before publishing.


5. **Verify the deploys:**
   - Native: EAS build finishes and the new build appears in TestFlight /
     Play Console internal track. Promote to the public tracks from there.
   - Web: `https://cardscentral.github.io/app/` and the landing page
     `https://cardscentral.github.io/` show the new version. (The Pages repo
     rebuilds immediately when `PAGES_DISPATCH_TOKEN` is configured, otherwise
     on its next scheduled run.)

## Hotfixes

For an urgent fix on an already-released version, branch from the tag, land the
fix on `main`, then repeat the steps above with the next PATCH version
(e.g. `v1.2.4`).

## Notes

- **Release notes:** with the current direct-commit-to-`main` flow, the notes
  generated by `--generate-notes` are a flat list of the commits since the last
  tag, so clear commit messages matter and a hand-written summary is worth it.
  [`.github/release.yml`](.github/release.yml) is already in place to group
  entries into labeled categories (Features, Bug Fixes, Shops, Translations, …)
  automatically — that grouping kicks in once the project moves to pull
  requests with labels.
- **Forks** without the `PAGES_DISPATCH_TOKEN` / EAS secrets simply skip the
  deploy jobs — no failures.
